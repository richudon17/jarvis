# JARVIS — “What changed since the one task right before?” (Verified Deltas)

## Summary
JARVIS is still the same small Python CLI agent, but the most recent meaningful changes are concentrated in **`core/orchestrator.py`** (placeholder/context resolution and how completion is summarized) and **`core/planner.py`** (stronger deterministic fallback plans for GUI/game/code-gen goals, plus a likely memory-prompt wiring issue).

## What changed (concrete code deltas)

### 1) Orchestrator placeholder resolution now has “previous vs latest” context + special `file_write` handling
**File**: `core/orchestrator.py`  
**Where**: `_previous_context`, `_latest_context`, `_is_placeholder_value`, `_replace_placeholders`, `_resolve_step_placeholders`

- The orchestrator computes:
  - `previous_context = _previous_context(step_results)`: concatenates **all** prior step results.
  - `latest_context = _latest_context(step_results)`: the **most recent** step result.
- Placeholder substitution depends on whether the value itself is a placeholder-like string.
- There is a special-case for `.py` writes:
  - When `tool == "file_write"` and the target path ends with `.py` and includes `"content"`,
    placeholder replacement is applied differently than for other tools.
- It also introduces a `prefer_latest` switch used when:
  - `tool == "file_write"` and the `previous_tool == "summarize_text"`

**Why this matters**: it changes what evidence the planner/tool receives. Previously described behavior (“string-based heuristics”) still exists, but now the *context injection strategy* is more nuanced and can reduce/alter context bleed depending on placeholder values.

---

### 2) Completion now triggers an evidence summary generated by the LLM
**File**: `core/orchestrator.py`  
**Where**: terminal handling block after `executed.get("status") == "done"`

- After `verify_goal(goal, completed, attempted)` passes:
  - it marks the goal completed
  - records episodic memory with `"success"`
  - and then **returns a synthesized summary**, not a fixed message:
    - builds `completed_summary` from the `completed` list
    - calls `_call_llm(summary_prompt, ...)`
    - falls back to a simple count-based summary on exception

**Why this matters**: “done” is still gated by verifier, but the final user-visible artifact is now generated from step evidence, which makes the run output more consistent with the stored history.

---

### 3) Planner now has deterministic 2-step fallback plans for GUI/game/code-gen goals
**File**: `core/planner.py`  
**Where**: `_is_gui_or_game_goal`, `_is_code_generation_goal`, `_raw_code_fallback_plan`

When the goal matches GUI/game or code-generation heuristics, planner skips the normal multi-tool JSON plan and instead returns:

1. `file_write` with complete code (either from local templates or LLM raw-code generation)
2. `done`

**Why this matters**:
- It reduces plan variability for these domains.
- It also aligns with tool constraints: those steps avoid `run_python` and other multi-step flows (especially because raw code needs to be written as a single complete artifact).

---

### 4) Planner has a likely prompt wiring bug: episodic memory context is constructed but overwritten
**File**: `core/planner.py`  
**Where**: `create_plan(...)`

- The code builds `episodes_text` from `memory.episodic.recall_recent(3)` and appends it into:
  - `user_prompt = f"{episodes_text}Goal: {goal}"`
- But immediately after, it performs:
  - `user_prompt = goal`
  - then adds `context` optionally

**Why this matters**: episodic memory recall likely does **not** influence the final LLM prompt as intended. This is a concrete “change” because it creates a functional gap between the documented/expected behavior and actual prompt content.

---

## Net effect for a developer
- You should treat **context/placeholder injection strategy** as a first-class behavior change in the orchestrator.
- You should expect **GUI/game/code-gen goals** to be handled via deterministic `file_write → done`.
- You should be aware that **episodic memory may not currently be used in planning** due to the `user_prompt` overwrite.
